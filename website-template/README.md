# Param-Table Template

A Hugo shortcode plus a data-file schema that standardizes scenario parameter
tables on the [krkn-chaos website](https://github.com/krkn-chaos/website), so the
documentation sync bot can keep them current by writing data only, never editing
human-written markdown.

This is the prerequisite the onboarding doc flagged ("scenario page template,
table structure?") and the "Hugo Data Files + Shortcode" item in the Core Pipeline.

## What is in here

```
website-template/
  layouts/shortcodes/param-table.html          # the shortcode (single source of truth)
  examples/data/params/<scenario>/<source>.yaml # worked example data files
```

The shortcode emits a plain table that inherits the site's existing table
styling (the same markup Docsy gives markdown tables), so no CSS changes are
needed in the website repo.

## How it works

A tab file renders its parameter table with one shortcode call:

```
{{< param-table scenario="node-scenarios" source="krkn-hub" >}}
```

The shortcode reads `data/params/<scenario>/<source>.yaml` and renders an HTML
table. Parameter and Description always render. Type, Possible Values, Default,
and Required render only when at least one row populates them (canonical schema
with auto-hide). If the data file is missing or empty, the shortcode renders
nothing and emits a Hugo `WARN`; the build stays green.

### Data file schema

```yaml
# Generated by krkn-docs-bot. Do not edit by hand.
source_repo: krkn-hub
source_ref: 9f3c1a2          # commit SHA the params were extracted from (provenance)
params:
  - name: ACTION             # required
    description: Action to run.   # required, one sentence, markdown allowed
    type: enum               # optional
    default: node_stop_start_scenario   # optional
    possible_values: [a, b]  # optional (krknctl)
    required: false          # optional (krknctl)
```

The bot extracts `name`, `type`, and `default` deterministically from the source;
only `description` is written by the LLM.

## One-time migration per tab file

The shortcode call is added to each tab file once, by hand, in the migration PR.
After that the bot only ever writes the data YAML.

Before (`_tab-krkn-hub.md`):
```
Parameter      | Description          | Type   | Default
-------------- | -------------------- | ------ | --------
ACTION         | Action to run        | enum   | node_stop_start_scenario
...
```

After:
```
{{< param-table scenario="node-scenarios" source="krkn-hub" >}}
```

All surrounding prose, run commands, and alerts stay untouched.

## Installing into the website repo

1. Copy `layouts/shortcodes/param-table.html` to the website's `layouts/shortcodes/`.
2. Add `data/params/<scenario>/<source>.yaml` files.
3. Replace each table region in `_tab-<source>.md` with the shortcode call.

No CSS changes are needed; the table inherits the site's existing styling.

## Running the tests

The tests build a minimal throwaway Hugo site and assert on the generated HTML
across the full edge-case matrix (column auto-hide, numeric-zero default,
required-false, possible_values, missing and empty data, markdown descriptions,
awkward default values, and the shipped example files).

```bash
npm install                              # provides hugo-extended
python -m pip install -r ../requirements-dev.txt
python -m pytest ../tests/ -v
```

Hugo comes from the `hugo-extended` npm devDependency, so no system Hugo install
is needed. The tests do not require Go or the Docsy theme.
